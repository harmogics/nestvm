# attributions — imprints of joint work

**Purpose.** The development side of the system's own attribution
obligation (Art. 14; Vol. 01 §7): the human's part traces through
commits; a model's part traces nowhere unless recorded. Each file here
is an **imprint** — a brief, factual record of one contribution arc,
naming who worked, in which roles, on what, and how.

**The paradigm this records.** In this repository code and documents are
written primarily *by models, in partnership*: the human holds vision,
direction, framing, challenges, the gates, and acceptance (Art. 9); a
model proposes, checks against the sources, implements, verifies, and
reports honestly. An imprint is the provenance of that partnership —
not a licence claim and not an assessment (judging quality is a separate
act, kept apart deliberately).

**Rules.**

1. One imprint per contribution arc, written by the model at the arc's
   close — the closing duty beside the landing records (PROTOCOL_DEV,
   the gate, step 6). If a session ends abruptly, the human may write it
   on the model's behalf, marked so.
2. **The model is identified exactly** — name, model id, harness. The
   human appears **by role**; the name is optional (commits carry it).
3. Brief and factual, honesty included: concessions, misses, and what
   reality checks caught belong in the imprint — they are part of how
   the work went.
4. Append-only: imprints are never edited; a correction is a new dated
   note inside the file or a successor imprint.
